<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| glTF/COLLADA output → user filesystem | Export writes to a user-chosen path; never to the mounted TRE archive or client install |
| Extract… → user filesystem | Writes raw bytes to user-chosen path; source is read-only TRE mount |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-19 | Tampering | ExportDialog — coordinate transform | mitigate | X-mirror applied on a scene CLONE (not the live scene) so the viewport is never corrupted by an export operation. Live scene geometry remains at original coordinates. |
| T-02-20 | Tampering | Extract… path traversal | mitigate | The destination path comes from OS showSaveFilePicker / Electron dialog.showSaveDialog (user-approved path only); the source bytes come from nativeCore.readMountEntry (never raw FS path open). No path concatenation with user-provided strings. |
| T-02-21 | Information Disclosure | glTF export includes all scene geometry | accept | This is intentional and user-initiated (the Export… button is explicit). The scene is a local asset the user loaded from their own TRE archive. No sensitive data exposure. |
| T-02-22 | Tampering | ColladaExporter output written as text | accept | ColladaExporter (Three.js bundled) produces XML text; XSS not a concern since output is saved to disk, not injected into the DOM. |
</threat_model>
