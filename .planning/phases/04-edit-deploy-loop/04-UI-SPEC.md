# Phase 4 — UI Design Contract: Edit & Deploy Loop

> Visual and interaction contract for the Phase 4 user-facing surfaces. Generated by gsd-ui-researcher.
> Consumed by gsd-planner (task decomposition), gsd-executor (implementation), gsd-ui-checker (validation), gsd-ui-auditor (retro audit).

**Scope (the new workflow surfaces this phase ships into the locked dockview shell — utilitarian IDE panels + one modal, NOT the 3D viewport):**
1. **Workspace entry + empty states** — IDE-style "open / create project folder" (D-04-01) and the "no workspace open" / "no client detected" empty surfaces (D-04-11).
2. **Staging panel** — the explicit "add to patch" list (D-04-02/03). Each row = a virtual path + replacement-file source + action (add / modify / delete-tombstone). Add/remove rows; shows exactly what ships in the next patch.
3. **Changeset timeline panel** — the Base44 layered changeset stack (D-04-05/06/07/08). Scrollable version list (v001, v002, …); active version highlighted; higher (rolled-back) layers greyed and re-activatable via a **non-destructive** active-version toggle; each row shows label, timestamp, #files changed, seal trigger.
4. **Deploy dialog** — modal: pick target client (auto-detected installs + manual override, D-04-09), choose deploy model (patch-prepend default vs shadow-base isolated opt-in with disk-space warning, D-04-10), show the chosen `.cfg` `searchTree_<sku>_<NN>=` slot (D-04-12), build+activate progress, success/failure with rollback/reset affordance. Deploy is **disabled** behind a clear "point me at a client" prompt when no client is set (D-04-11).
5. **VCS panel** — Git/LFS status for the workspace (D-04-13/15/16): commit message + commit/push, live log feed, LFS status, and a clear surface for the pre-commit retail-bytes guard outcome.

**Out of scope (do NOT design here — inherited locks, or other phases):**
- The app shell, titlebar, theme picker, activity bar, panel chrome, docking, status bar — **LOCKED by sketch 001 + `00-UI-SPEC.md`**. This spec consumes those tokens; it never redefines them.
- The 3D viewport, render chrome, gizmo, customization/material/animation panels — **shipped and locked in `02-UI-SPEC.md`**. This phase adds editor/deploy panels alongside them; it does not touch the viewport.
- Typed/rich in-app editors (DTII grid, `.stf`, transform gizmo into the running client) — **Phase 5**. Edits in this phase arrive via Extract→edit-externally / drop-in (D-04-03); the in-app IFF view stays read-only (Phase 1 D-08).
- v6000 (encrypted) archives — enumerate-only, never repacked/patched (Phase 1 lock).
- Remote differential CDN sync (version-control doc layer 4) — v2/deferred.

**Inheritance:** Every token, accessibility rule, copy rule, and registry decision below is inherited verbatim from `00-UI-SPEC.md` and extended by `01-`/`02-UI-SPEC.md`, unless a Phase-4 addition is explicitly called out. New rows (icon-only controls, copy strings, panel states) EXTEND the prior exhaustive lists — they do not replace them. The reference React components shipped in `docs/06-workflow/version-control-and-backup.md` (`SwgChangesetTimelinePanel`, `SwgGitVersionControlPanel`, deploy/cfg widgets) are **layout starting points only**; their inline neon styles are explicitly REJECTED — bind to the tokens below, not to the doc's styles.

## Spacing Scale — 3px DCC base grid (project grid exception, inherited)

This project deliberately overrides the standard 4px/8px web grid with a **3px DCC base unit** to match Blender-style panel density (validated in sketch 001, locked in `00-UI-SPEC.md`). All spacing tokens are multiples of the 3px project base. This is a documented, intentional project-level exception — the gsd-ui-checker must evaluate spacing against this 3px grid, **NOT** a 4px or 8px grid.

Use these tokens exclusively. Do not use arbitrary pixel values.

| Token | Value | Phase-4 usage |
|-------|-------|---------------|
| `--space-1` | 3px | Action-badge internal padding, status-glyph gaps, list-row inner gap |
| `--space-2` | 6px | Panel-head padding, staging/changeset row item spacing, button gaps, field-row gaps |
| `--space-3` | 9px | List-row vertical padding, dialog section-title spacing, log-feed line padding |
| `--space-4` | 12px | Panel body padding, dialog body inset, commit-message textarea padding |
| `--space-6` | 18px | Inter-section spacing inside the deploy dialog (client / model / slot blocks) |
| `--space-8` | 24px | (reserved — major layout gaps) |

**Exceptions (inherited chrome dimensions / structural values — not spacing values):**
- `--tabstrip-h: 30px` — panel head height (every Phase-4 panel head matches `LiveInspectorPanel`/`InspectorPanel`). Reuse for the staging/changeset/VCS panel heads.
- Virtualized list `ROW_HEIGHT: 30px` (the established `VfsTree` row height; `OVERSCAN: 8`) — **mandatory** for the staging list and changeset timeline (they can grow long). Rows are a fixed 30px regardless of selection so virtualization math holds (inherited `VfsTree` contract).
- Deploy dialog width: 360px (modal; `ExportDialog` is 320 — the deploy dialog carries more fields, so 360, `maxWidth: 90vw`), `--radius-md` corners, `0 8px 32px rgba(0,0,0,0.6)` shadow — verbatim `ExportDialog` modal pattern.
- `--statusbar-h: 22px`, `--activity-w: 46px` — fixed inherited chrome.

## Typography

Inherited from `00-UI-SPEC.md` — two font families, four sizes, weights **400 / 600 only**. Monospace is **mandatory** for all paths, hashes, timestamps, byte sizes, slot numbers, commit SHAs, version labels, and config-key strings (this phase is path-and-identifier heavy).

| Role | Token | Size | Weight | Line Height | Font | Phase-4 usage |
|------|-------|------|--------|-------------|------|---------------|
| Micro / technical | `--text-xs` | 11px | 400 | 1.5 | **mono** | Virtual paths, replacement-file paths, timestamps, `#files` counts, byte sizes, `searchTree_<sku>_<NN>=` key, version labels (`v001`), commit SHAs, disk-space estimates, git log lines, LFS pointer counts |
| Label / control | `--text-sm` | 12px | 400 | 1.4 | sans | Action-badge labels (add/modify/delete), button labels, changeset row label text, deploy-model radio labels, field labels, panel-head titles |
| Body / field | `--text-base` | 13px | 400 | 1.5 | sans | Empty-state body, warning/error body, dialog prose, confirmation copy |
| Subheading / name | `--text-md` | 14px | 600 | 1.3 | sans | Panel/section heads ("Staging", "Changesets", "Deploy", "Version Control"), dialog title, empty-state heading, error heading |

**Phase-4 mono mandate (extends the inherited rule):** virtual paths, on-disk replacement paths, timestamps (`2026-06-26 15:30`), file counts, byte/disk sizes, the literal `.cfg` key (`searchTree_00_55=swgtoolkit_mymod.tre`), `maxSearchPriority` value, version indices, commit short-SHAs, and branch/remote names render in **mono**. SWG 4-byte format tags shown inline (e.g. `EERT5000`) render in mono (inherited from `01-UI-SPEC.md`). Sans for prose: panel heads, button labels, empty-state copy, warning/error sentences, dialog body.

## Color

Inherited 60/30/10 split from `00-UI-SPEC.md`. Theme-driven; cyan is the active default and re-styles automatically across all five themes (incl. high-contrast). Accent stays **scarce** — it marks selection/active/success only, never decoration. State is **never** conveyed by color alone (Accessibility Rule 1: glyph + color + text).

| Role | Hex (default/cyan theme) | Token | Phase-4 usage |
|------|--------------------------|-------|---------------|
| Dominant (60%) — deep background | `#181818` | `--color-bg` | Shell frame, panel gaps, input/textarea field background |
| Secondary (30%) — panel surfaces | `#242424` body / `#323232` head | `--color-surface` / `--color-header` | Staging / changeset / VCS panel bodies, panel heads, deploy dialog surface |
| Nested rows / hover | `#2b2b2b` | `--color-surface-2` | Staging row hover, changeset row hover, log-line hover |
| Active layer fill | `rgba(56,190,201,0.16)` | `--color-accent-dim` | Active-changeset row background (paired with 2px accent left border) |
| Widget / control | `#3a3a3a` / `#474747` hover | `--color-widget` / `--color-widget-hover` | Disabled primary button fill, progress track, secondary control surfaces |
| Accent (10%) — hologram cyan | `#38bec9` | `--color-accent` | See reserved-for list below |
| Success | `#38bec9` (accent) | `--color-accent` | Deploy succeeded ✓, patch verified loads/shadows ✓, commit succeeded ✓, guard passed ✓ |
| Warning | `#e0a13a` | `--color-warn` | Delete-tombstone badge, shadow-base disk-space warning, git-lfs-absent notice, `maxSearchPriority` bump notice, "no client detected" deploy-disabled prompt |
| Destructive / error | `#e0584f` | `--color-danger` | Pre-commit retail-bytes guard FAIL, deploy build/activate failure, reset-deployment confirmation accent |
| Info | `#4a8cff` | `--color-info` | Add-new staging badge, neutral notes (chosen slot, detected client path, patch version `5000`, sealedBy caption) |

**Accent reserved for (Phase-4 additions to the prior exhaustive lists — add nothing else):**
1. **Active changeset row** — the current `activeVersionIndex` layer: 2px accent left border + `--color-accent-dim` background + `● active` pip (the inherited selection pattern).
2. **Primary CTA fill** — the dialog's confirm action (`Deploy patch`, `Pack Patch`) and the staging `Add to Patch` primary button use `background: --color-accent; color: --color-accent-text` when enabled (inherited `primaryBtnStyle` from `ExportDialog`). Disabled → `--color-widget` fill, `--color-text-faint` text, `opacity 0.6`.
3. **Success glyphs/captions** — `✓ deployed`, `✓ patch loads & shadows`, `✓ committed`, `✓ guard passed` (via `VerificationStatus variant="pass"`, which already maps `pass → --color-accent`).
4. **Focus ring** on all interactive controls (`--focus-ring`) — inherited.
5. **Selected deploy-model radio** — the active model option (patch-prepend / shadow-base) gets a 2px accent ring + `--color-accent-dim` fill (selection signal, paired with the radio dot).

**Accent must NOT appear on:** action badges (use info/warn/text), rolled-back/greyed changeset layers, git log lines, disabled controls, panel borders, the progress track, decorative dividers, or any non-interactive background. Semantic states (warn/error/info) use their OWN token, never accent.

> **Field-background note:** `LiveInspectorPanel` references an **undefined** `--color-input` token (falls back to transparent — a latent bug). Phase-4 inputs/textareas MUST use a **defined** token: `background: --color-bg`, `border: 1px solid --color-border`, `--radius-sm`. Do not propagate `--color-input`.

## Surface 2 — Staging Panel ("add to patch" list)

D-04-02 / D-04-03 — the explicit list that decouples "edited" from "will ship". Each entry maps 1:1 onto `buildTre`'s `{path, data | tombstone}` shape.

- **Panel head (30px):** `Staging` (`--text-md`, weight 600) + a right-aligned `Pack Patch` action (primary; auto-seals a changeset per D-04-07) and an `Add…` action.
- **Add affordances:** (a) `Add from Extract` — stage the currently-extracted asset (virtual path + the on-disk replacement written by Phase 2 Extract); (b) `Add file…` — pick a replacement file and type/confirm its virtual path (add-new or modify); (c) `Add tombstone…` — pick a virtual path to delete (hide retail). Drag-drop a file onto the panel = `Add file…` prefilled.
- **Virtualized list (mandatory, `ROW_HEIGHT 30`, `OVERSCAN 8`):** one row per staged entry. Row anatomy (single line, fixed 30px):
  - **action badge** (left): glyph + label — `+ add` (`--color-info`), `~ modify` (`--color-text-muted`), `⊘ delete (tombstone)` (`--color-warn`). Triple-encoded (Rule 1).
  - **virtual path** (mono `--text-xs`, ellipsis-middle, primary text) — the in-archive path the entry resolves to.
  - **source** (mono `--text-xs`, `--color-text-faint`, right-aligned): replacement filename + byte size for add/modify; `(tombstone — length-0)` for delete.
  - **remove control** (`×`, icon-only, `aria-label="Remove from patch"`) — removing a row is **non-destructive** (drops it from the staging list only; never deletes the on-disk replacement file).
- **Row hover:** `--color-surface-2` background (inherited 0.1s ease).
- **Footer summary:** `{n} staged · {a} add · {m} modify · {d} delete` (mono `--text-xs`, `--color-text-faint`) + total payload size. This is the "what will ship in the next patch" readout.

### States

| State | Content |
|-------|---------|
| Empty (nothing staged) | Centered: heading `Nothing staged` (`--text-md`, `--color-text-muted`) + body `Extract a file and Add to patch, or drop in a replacement.` (`--text-base`, `--color-text-faint`). `Pack Patch` disabled. |
| Has entries | Virtualized rows + footer summary. `Pack Patch` enabled. |
| Replacement file missing on disk (was moved/deleted externally) | The row shows `⚠ source missing` (`--color-warn`) on the source column + a re-link affordance (`Relink…`); `Pack Patch` disabled until resolved (a tombstone-only patch is exempt). |
| Path-traversal rejected | If a virtual path contains `..`/absolute escape, reject inline with `✕ invalid path — `..` not allowed` (`--color-danger`); the row is not added (Security V12). |

## Surface 3 — Changeset Timeline Panel (Base44 layered stack)

D-04-05/06/07/08 — the layered changeset stack with a **non-destructive** active-version toggle. The maintainer's experience-backed model: rollback moves the active-version pointer DOWN; higher layers persist on disk, render greyed, and are re-activatable (redo). Nothing is ever deleted on rollback.

- **Panel head (30px):** `Changesets` (`--text-md`, weight 600) + a right-aligned `Commit Changeset` action (manual checkpoint, D-04-07).
- **Virtualized list (mandatory if long, `ROW_HEIGHT 30`, `OVERSCAN 8`):** newest layer at top, oldest at bottom. One row per sealed layer. Row anatomy (fixed 30px):
  - **version label** (mono `--text-xs`, e.g. `v003`) + **state pip**: `● active` (accent) for `activeVersionIndex`; `↑ rolled back` (`--color-text-faint`) for any layer above the active pointer.
  - **label/message** (`--text-sm`, sans, primary text, ellipsis) — the changeset's human label.
  - **metadata** (mono `--text-xs`, `--color-text-faint`, right-aligned): timestamp + `{n} files` + seal-trigger badge (`◷ checkpoint` manual / `⤓ auto (pack)`).
  - **Active layer styling:** 2px accent left border + `--color-accent-dim` background (the inherited selection pattern).
  - **Rolled-back layers:** entire row dimmed to `--color-text-faint`; visually below-the-fold of the active pointer in saturation, NOT hidden — they stay clickable to re-activate.
- **Toggle interaction (the core contract):** clicking any row sets it as the active version (writes `activeVersionIndex`, a JSON pointer move — never a native delete). Clicking a lower row = **rollback**; clicking a higher greyed row = **redo/re-activate**. Both are instant, non-destructive, and re-flatten the workspace VFS top-down. No confirmation dialog (work is never lost — D-04-08); a brief status-bar confirmation `active version → v00N` suffices.
- **Banned (do NOT build):** any `PurgeChangesetLayer` / destructive delete, any second `.tar.gz` snapshot history (RESEARCH Pitfall 6). One history system (D-04-13).

### States

| State | Content |
|-------|---------|
| Empty (no changesets) | Centered: heading `No changesets yet` (`--text-md`, `--color-text-muted`) + body `Commit a changeset or pack a patch to create your first rollback point.` (`--text-base`, `--color-text-faint`). |
| Single layer (v001 active) | One active row; no rolled-back rows. |
| Multiple, active at top | All layers active-or-below; no greyed rows. |
| Rolled back (active below top) | Layers above the active pointer render greyed with `↑ rolled back · re-activatable`; the active row carries the accent border. |
| Commit-changeset prompt | A lightweight inline label field (`Changeset label…` placeholder, mono) at the head when `Commit Changeset` is pressed; Enter seals the layer. |

## Surface 4 — Deploy Dialog (modal)

DEPLOY-02 — pick client, pick deploy model, show the cfg slot, build+activate, report. **Verbatim structural reuse of `ExportDialog.tsx`** (overlay + 360px panel + header with `×` + dividers + `AsyncProgress` during work + `VerificationStatus` on done/fail + retry). This is the phase's primary CTA surface.

- **Trigger:** a `Deploy…` action (in the staging panel head or status bar). **Disabled** when no client is set, with the `⚠ No client selected — point me at a SWG client to deploy.` prompt inline (D-04-11) — mirrors `ExportDialog`'s disabled-primary pattern.
- **Section A — Target client (D-04-09):**
  - Auto-detected installs listed as selectable rows: client name + install path (mono `--text-xs`, `--color-info`) + detected TRE version (`EERT5000`, mono). Radio-select one.
  - `Browse…` for a **manual folder override** (the non-blocking escape hatch — manual override always available even if auto-detect finds nothing).
  - If none detected: `No installs auto-detected — choose a client folder.` (`--color-text-muted`) + `Browse…`.
- **Section B — Deploy model (D-04-10):** two radio options (active option = accent ring + `--color-accent-dim`):
  - **Patch-prepend** (default, selected): caption `adds the patch at a free higher cfg priority — retail files stay pristine` (`--text-base`).
  - **Shadow-base (isolated client)** (opt-in): caption `copies the client TRE base to a local shadow and patches there`. Selecting it reveals a `⚠` disk-space warning with the estimated copy size (mono, e.g. `~3.4 GB`) — `--color-warn`. The shadow is local-only, never git-tracked.
- **Section C — Config slot preview (D-04-12):** read-only mono block showing exactly what will be written: the resolved `[SharedFile]` target, the chosen `searchTree_{sku}_{NN}=patch.tre` key, and the `maxSearchPriority` check. Example:
  - `[SharedFile]` / `searchTree_00_55=swgtoolkit_mymod.tre` / `maxSearchPriority=60 ✓ (slot 55 ≤ 60)`.
  - If the chosen slot exceeds `maxSearchPriority` (e.g. stock SWGEmu default 20), show `⚠ will bump maxSearchPriority {old} → {new}` (`--color-warn`).
  - Note line (mono, `--color-info`): `writes a toolkit-owned cfg via .include — never touches user.cfg/options.cfg`.
- **Action row:** `[Cancel] [Deploy patch]` — `Deploy patch` is the primary (accent) CTA; disabled until a client + model are chosen.
- **Build → activate sequence:** on confirm, show `AsyncProgress` `Building patch (v5000)…` then `Writing client config…`; the dialog's buttons disable during work.
- **Success:** `VerificationStatus variant="pass"` → `✓ deployed · slot _{sku}_{NN}` + a `Reset deployment` text action (removes the one cfg key + deletes the deployed patch; changesets are kept) and a `Reveal patch` action.
- **Failure (build):** `✕ Could not build patch — {reason}` (`--color-danger`) + `Retry`; dialog stays open.
- **Failure (activation):** `✕ Could not write client config — {reason}. The .cfg was restored from backup.` (`--color-danger`) + `Retry`. The backup-restore guarantee (D-04-12) is stated in the copy so the user knows the client config is intact.

### States

| State | Content |
|-------|---------|
| No client set (trigger) | `Deploy…` disabled; `⚠ No client selected` prompt. |
| Dialog open, choosing | Sections A/B/C; `Deploy patch` disabled until client + model chosen. |
| Shadow-base selected | Disk-space `⚠` warning with estimate revealed. |
| Building / activating | `AsyncProgress` sweep; buttons disabled. |
| Deployed | `✓ deployed · slot` + `Reset deployment` + `Reveal patch`. |
| Build/activate failed | `✕` danger status + `Retry`; backup-restore note on activation failure. |
| Reset confirm (destructive — external mutation) | `Reset deployment? This removes the patch from the client config and deletes the deployed patch .tre. Your changesets are kept.` + `[Cancel] [Reset]` (Reset uses `--color-danger`). |

## Surface 5 — VCS Panel (Git / LFS)

DEPLOY-04 / D-04-13/15/16 — Git is the collaboration backbone *under* the changeset UX (one history, not two). Surfaces commit/push, the log feed, LFS status, and — critically — the pre-commit retail-bytes guard outcome.

- **Panel head (30px):** `Version Control` (`--text-md`, weight 600) + branch/remote indicator (mono `--text-xs`, `--color-text-faint`).
- **Changes-to-commit list:** the explicitly-staged `.studio/` paths (D-04-15 — never blind `git add .`). Each line: path (mono `--text-xs`) + change glyph (`+`/`~`/`⊘`, info/text/warn — same encoding as staging). Read-only; the changeset commit drives what's staged.
- **Commit message field:** a sans textarea (`--text-base`, background `--color-bg`, border `--color-border`, `--radius-sm`, `--space-4` padding) with placeholder `Describe this changeset…`. Below it: `[Commit]` (primary) and `[Commit & Push]` (secondary). Commit messages flow to `execFile('git', ['commit','-m', msg])` — **never** string interpolation (D-04-16 / Security: command-injection vector).
- **Pre-commit guard surface (the standout requirement):** the guard outcome is surfaced **prominently** on every commit attempt via `VerificationStatus`:
  - Pass → `✓ guard passed — mod outputs only` (accent).
  - Fail → `✕ blocked: {file} looks like retail/.tre bytes — never commit a patch or retail archive` (`--color-danger`), with the offending path (mono) and the reason (size > 50 MB, `.tre` extension, or retail fingerprint). The commit is refused; the message field stays populated so the user doesn't lose their text.
- **LFS status line:** `git-lfs {version} · {n} pointers` (mono, `--color-text-faint`) when present; `⚠ git-lfs not found — large binaries will bloat history. Install git-lfs to enable LFS routing.` (`--color-warn`) when absent (D-04-16 — check presence before LFS ops).
- **Live log feed:** a scrollable list of recent commits — `{shortSha} {subject} · {relative-time}` (mono `--text-xs`, hover `--color-surface-2`). Newest first. Virtualize if it grows long.

### States

| State | Content |
|-------|---------|
| No remote configured | `Commit & Push` → `Commit` only; a `Set remote…` affordance; `no remote` (mono, faint). |
| Working tree clean | Changes list empty: `Working tree clean` (`--color-text-muted`); `Commit` disabled. |
| Has staged changes | Changes list + enabled `Commit` / `Commit & Push`. |
| Guard blocked | Danger `VerificationStatus` + offending path; commit refused; message preserved. |
| git-lfs absent | `⚠` LFS warning line; commits still allowed (binaries warned). |
| Push in progress | `AsyncProgress` `Pushing to {remote}…`; buttons disabled. |
| Push failed | `✕ Push failed — {reason}.` (`--color-danger`) + `Retry`. |

|--------|--------------|--------------|
