<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| commit message textarea → execFile git commit | User-typed commit message flows into git argv; must never allow shell metacharacters to escape argument isolation |
| staged file path list → execFile git add | File paths in stagePaths flow into git argv; must be relative workspace-relative paths |
| pre-commit hook output → UI | Hook stderr is displayed to user; must not expose sensitive paths |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-23 | Tampering | gitCommit message → execFile argv | mitigate | execFile isolation means argv[N] is passed as a literal argument, NOT shell-interpolated; strip null bytes and control chars from message (defense-in-depth); never use exec() (D-04-16) |
| T-04-24 | Tampering | stagePaths containing '../' escape | mitigate | Each path in stagePaths must be relative and workspace-relative; validate no '..' prefix in gitCommit before execFileAsync |
| T-04-25 | Information Disclosure | copyright leak — retail bytes committed | mitigate | Defense-in-depth D-04-15: .gitignore *.tre (workspaceService), explicit-path staging (never git add .), pre-commit hook (workspaceService installs), app-side getGuardStatus check before every commit |
| T-04-26 | Denial of Service | large binary committed and bloating clone | mitigate | Pre-commit hook size guard (> 50 MB → reject); app-side getGuardStatus also checks size |
| T-04-SC | Tampering | npm/pip installs | mitigate | No new npm packages; slopcheck not required |
