## Phase Goal

**As a** SWG mod developer, **I want to** stage edited files in a project workspace and build a deployable `.tre` patch that activates via the client config, **so that** I can iterate on mod changes in-game, roll back to any prior state, and version my work safely via Git/LFS.

<objective>
Build the DeployDialog modal (the "Deploy…" CTA) and prove the full end-to-end loop with an in-client UAT on SWG Infinity: stage a file, pack a v5000 patch, activate via .include'd swgtoolkit.cfg at slot ≤ 60, verify the mod loads in-game, confirm persistence across launcher relaunch (OQ-2/A2), then roll back.

DeployDialog also branches on deployModel: when the user selects 'shadow-base', handleDeploy calls deployShadowBase (created in 04-06b) instead of the patch-prepend cfgActivator sequence. Both paths converge on sealLayer('pack') auto-seal.

Output: DeployDialog.tsx (modal: A=client, B=model, C=slot preview, build/activate phases, success/fail/reset, deployModel dispatch); manual UAT checkpoint approved on SWG Infinity.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DeployDialog → client filesystem | Dialog copies patch.tre into the client Live/ dir and writes swgtoolkit.cfg |
| user-confirmed installPath → cfgActivator | The installPath confirmed by user selection gates where the patch and cfg writes go |
| sealLayer deployRecord → .studio/changesets | The deployed patch path is stored in the sealed changeset; must be a valid resolvable path |
| deployShadowBase → shadow copy dir | Shadow copy goes to .studio/shadow/ — must stay inside workspace, never git-tracked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-20 | Tampering | DeployDialog patch copy to wrong client dir | mitigate | D-04-11: installPath is user-confirmed via detectClients() result or explicit Browse… override; validate path.isAbsolute and that swgemu.cfg exists at cfgRootPath before deploying |
| T-04-21 | Tampering | cfg write failure leaves partial state | mitigate | activatePatch is atomic (backup + tmp+rename); on failure deactivatePatch restores the backup; the error message states 'restored from backup' so user knows the cfg is intact |
