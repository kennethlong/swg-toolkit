# Phase 1 — UI Design Contract: Core Engine — IFF + TRE + Verification Harness

> Visual and interaction contract for the Phase 1 user-facing surfaces. Generated by gsd-ui-researcher.
> Consumed by gsd-planner (task decomposition), gsd-executor (implementation), gsd-ui-checker (validation), gsd-ui-auditor (retro audit).

**Scope (the four new surfaces this phase ships into the Phase-0 dockview shell):**
1. **TRE virtual-filesystem (VFS) browser** — mount one+ `.tre` archives, browse + search by path/name, and **communicate which archive each node resolves from** in the load-order/override (shadow) chain (CORE-01, CORE-02; D-06).
2. **IFF FORM/chunk navigable tree viewer** — parse an arbitrary real IFF into a navigable structure tree (tag / size / byte-offset / kind), surface "zero unexplained trailing bytes" and round-trip byte-exact status (CORE-03; D-06/D-07).
3. **Hex/ASCII inspector pane** — `offset │ hex │ ascii` view of the selected leaf chunk's raw bytes. The SIE-successor baseline; the surface typed editors plug into later (D-07).
4. **Async / progress + verification-status affordances** — long mount/decompress/parse of multi-GB archives runs off-main-thread (`Napi::AsyncWorker`); the UI must stay responsive and show progress without main-thread freeze (CORE-06), and surface the byte-exact round-trip gate state (CORE-04/CORE-05).

**Out of scope (do NOT design here):**
- The app shell, titlebar, theme picker, panel chrome, docking, status bar — **all LOCKED by sketch 001 + `00-UI-SPEC.md`**. This spec consumes those tokens; it does not redefine them.
- 3D viewport / mesh rendering — Phase 2.
- **In-UI IFF value editing with save-back** — D-08: Phase 1 proves the write path through the harness/tests only, NOT the UI. This spec defines no editable IFF fields, no "Save" affordance, no dirty-cell editor for IFF.
- Per-format typed decoders (datatable grid, STF, mesh) — Phases 2/5/7.
- `.tre` patch-packaging / `.cfg` activation user workflow — Phase 4 (D-04 ships the builder primitive, not the deploy UI).

**Inheritance:** Every token, accessibility rule, copy rule, and registry decision below is inherited verbatim from `00-UI-SPEC.md` unless a Phase-1 addition is explicitly called out. Where this spec adds rows (new icon-only controls, new copy strings, new tree-node states) they EXTEND the Phase-0 exhaustive lists.

## Spacing Scale — 3px DCC base grid (project grid exception, inherited)

This project deliberately overrides the standard 4px/8px web grid with a **3px DCC base unit** to match
Blender-style panel density (validated in sketch 001, locked in `00-UI-SPEC.md`). All spacing tokens are
multiples of the 3px project base unit. This is a documented, intentional project-level exception — the
gsd-ui-checker must evaluate spacing against this 3px grid, NOT the default 4px grid.

Use these tokens exclusively. Do not use arbitrary pixel values.

| Token | Value | Phase-1 usage |
|-------|-------|---------------|
| `--space-1` | 3px | Icon-to-label gap in tree rows; hex/ascii inner cell padding; shadow-chain pip gap |
| `--space-2` | 6px | Tree-row vertical padding; search-field internal padding; inline status-glyph gap; toolbar item spacing |
| `--space-3` | 9px | Tree-section head padding; mount-toolbar vertical padding |
| `--space-4` | 12px | Panel body padding; hex-pane horizontal inset; empty-state inset |
| `--space-6` | 18px | Inter-section spacing (e.g. between mount toolbar and tree) |
| `--space-8` | 24px | (reserved — major layout gaps) |

**Exceptions (inherited chrome dimensions — not spacing values):**
- `--tabstrip-h: 30px`, `--statusbar-h: 22px`, `--activity-w: 46px` — fixed chrome dimensions.
- Tree indentation: **16px per nesting level** via `padding-left` (inherited from `00-UI-SPEC.md` Assets tree). Applies to BOTH the VFS path tree and the IFF FORM tree.
- Hex inspector row height: **18px** fixed (= `--space-6`) for monospace byte-grid alignment.

## Typography

Inherited from `00-UI-SPEC.md` — two font families, four sizes, weights 400 / 600 only. Monospace is
**mandatory** for all byte/offset/size/path-technical content (this phase is byte-centric).

| Role | Token | Size | Weight | Line Height | Font | Phase-1 usage |
|------|-------|------|--------|-------------|------|---------------|
| Micro / technical | `--text-xs` | 11px | 400 | 1.5 | **mono** | Hex/ASCII bytes, byte-offsets, chunk sizes, CRC values, archive badges, progress %, status-glyph captions |
| Label / tree | `--text-sm` | 12px | 400 | 1.4 | sans | Tree row labels (path segments, FORM/chunk tags shown as labels), search placeholder, tab labels |

**Phase-1 rules (extend Phase-0 mono mandate):**
- **Monospace is mandatory** for: hex byte columns, ASCII gutter, byte-offset column, chunk `tag`/`size`/`offset` values in the IFF tree, TRE TOC values (CRC, compressed/uncompressed size, compressor), search match counts, progress percentages, and the resolved-archive name shown in a node's shadow tooltip.
- **IFF FORM/chunk 4-character tags** (e.g. `FORM`, `DERV`, `0000`) render in **mono** even when inline in a sans tree row, because they are literal 4-byte tokens, not prose.

## Color

Inherited 60/30/10 split from `00-UI-SPEC.md`. Theme-driven; cyan is the active default. Accent stays
scarce — it marks selection/active/success only, never decoration. State is **never** conveyed by color
alone (Accessibility Rule 1).

| Role | Hex (default/cyan theme) | Token | Phase-1 usage |
|------|--------------------------|-------|---------------|
| Dominant (60%) — deep background | `#181818` | `--color-bg` | Shell frame, gaps |
| Secondary (30%) — panel surfaces | `#242424` body / `#323232` header | `--color-surface` / `--color-header` | VFS browser body, IFF tree body, hex pane bg, section heads |
| Nested rows / hover / hex even-column | `#2b2b2b` | `--color-surface-2` | Tree row hover, alternating hex-row shade, nested TOC detail rows |
| Accent (10%) — hologram cyan | `#38bec9` | `--color-accent` | See reserved-for list below |
| Success | `#38bec9` (accent) | `--color-accent` | Round-trip PASS, "byte-exact ✓", "0 trailing bytes ✓", mount-complete ✓ |
| Warning | `#e0a13a` | `--color-warn` | v0006 enumerate-only banner, "unknown trailing bytes present", non-byte-exact-capable archive |
| Destructive / error | `#e0584f` | `--color-danger` | Round-trip FAIL, parse error, mount failure, malformed-IFF marker |
| Info | `#4a8cff` | `--color-info` | Neutral informational notes (e.g. "compressor: raw-deflate") |

**Accent reserved for (Phase-1 additions to the Phase-0 exhaustive list — add nothing else):**
1. Selected tree row left border (2px solid accent) + `--color-accent-dim` background — applies to **both** the VFS path tree and the IFF FORM tree. *(inherited pattern)*
2. Focus ring on all interactive controls (`--focus-ring`). *(inherited)*
3. **Round-trip / verification PASS** glyph and "byte-exact ✓" / "0 trailing bytes ✓" tokens.
4. **Mount-complete ✓** and **"resolves from {archive}"** wins-indicator pip in the shadow chain.
5. Selected hex-byte highlight: the byte range corresponding to the selected IFF node header is highlighted with `--color-accent-dim` background in the hex pane (so selecting a chunk shows its bytes).
6. Search-match highlight: matched substring in a tree label gets `--color-accent-dim` background + `--color-accent` text. *(NEVER color alone — the row also remains a normal tree row; the highlight is on the matched run.)*
7. Active progress-bar fill (`--color-accent` on `--color-widget` track).

**Accent must NOT appear on:** unselected rows, inactive tabs, decorative borders, the hex grid lines, disabled controls, or any non-interactive background. Semantic states (warn/error/info) use their OWN semantic token, not accent.

### Semantic-state color is always paired with a glyph (Accessibility Rule 1 extension)

Verification/parse status is conveyed by **glyph + color together**, never color alone:

| State | Glyph | Color token | Label (mono, `--text-xs`) |
|-------|-------|-------------|---------------------------|
| Byte-exact round-trip PASS | `✓` | `--color-accent` | `byte-exact ✓` |
| Round-trip FAIL | `✕` | `--color-danger` | `round-trip FAIL @ 0x{offset}` |
| Structural-only (v0006 encrypted) | `≈` | `--color-warn` | `enumerate-only (encrypted)` |
| Zero unexplained trailing bytes | `✓` | `--color-accent` | `0 trailing bytes ✓` |
| Trailing bytes present (surfaced, not dropped) | `▴` | `--color-warn` | `{n} trailing bytes` |
| Parse error / malformed | `✕` | `--color-danger` | `parse error` |

## Surface 1 — TRE Virtual-Filesystem Browser (LEFT / Assets panel)

Replaces the Phase-0 "No archive mounted" seed state in `SidebarPanel.tsx` with a functional mount +
browse + search + shadow-resolution surface. Lives in the existing LEFT dock region.

### Layout (top → bottom, flex column on `--color-surface`)

1. **Mount toolbar** (head row, `--tabstrip-h`-style strip on `--color-header`):
   - `[+ Mount Archive…]` action — opens the OS file picker for one or more `.tre` files.
   - Mounted-archive count chip: e.g. `3 archives` (`--text-xs`, mono, `--color-text-faint`).
   - Overflow action: `⋮` (aria-label "Mount options") — reserved for reorder/unmount (see below).
2. **Mounted Archives list** (the load-order / override chain — the keystone of CORE-01):
   - Section head: `Mounted Archives` (`--text-md`, weight 600).
   - One row per archive, **in resolution priority order, highest-priority first** (RESEARCH: priority-sorted search-node list; first-match-wins shadow).
   - Row anatomy: priority index badge (`#1`, mono, `--color-text-faint`) · archive filename (`--text-sm`) · version tag chip (`v0005` / `v0006`, mono `--text-xs`) · entry count (mono `--text-xs`, `--color-text-faint`, right-aligned).
   - **Override affordance:** a row may be dragged to reorder priority; on reorder, the VFS tree below re-resolves. (Drag handle ⠿ at row left, inherited handle styling.) If reorder is deferred to a later wave, the order still reflects the cfg/mount order and the priority index must be shown.
   - v0006 rows carry the `≈ enumerate-only (encrypted)` warn chip (payloads are encrypted — D-05).
3. **Search field** (CORE-02):
   - Full-width input, `--color-widget` bg, mono placeholder `Search path or name…`.
   - Default semantics: **case-insensitive substring** over the flat name list (RESEARCH-resolved). A small toggle `[*]` (aria-label "Glob match") switches to glob (`*`/`?`). Regex is NOT offered in Phase 1.
   - Live match count to the right: `{n} matches` (mono, `--text-xs`).
   - Search runs **native-side returning matched indices** (archives can hold 100k+ entries — do not ship the full name list per keystroke; RESEARCH).
4. **VFS path tree** (the resolved virtual filesystem):
   - Folder/file tree of the merged, shadow-resolved namespace. 16px indent per level. Tree-row states inherited from `00-UI-SPEC.md` (hover `--color-surface-2`; selected 2px accent left border + `--color-accent-dim`).
   - Chevron `▶` (rotates) for folders; file rows show a type glyph + label.

### Shadow / override resolution — the must-communicate contract (CORE-01)

The UI **must** make it unambiguous which archive a node resolves from and what it shadows:

- **Per-file resolved-archive indicator:** every file row shows, right-aligned, a small archive pip + the winning archive's short name (mono `--text-xs`, `--color-text-faint`); e.g. `mtg_patch_03`. When that file is **overriding** a lower archive (shadowing), the pip is rendered in `--color-accent` (the win-indicator from the accent list) and the row carries a `⧉` shadow glyph (aria-label "Overrides {n} lower archive(s)").
- **Shadow-chain tooltip / detail:** hovering (or selecting) a shadowed file reveals the full chain, highest-first, e.g.
  ```
  resolves from:  mtg_patch_03.tre        ✓ wins
  shadows:        infinity_custom_07.tre
  shadows:        bottom.tre
  ```
  Winner line uses `✓ wins` in `--color-accent`; shadowed lines in `--color-text-muted` with a strikethrough-free `shadows:` label (do not strike text — use the label, per "no color/effect-alone" discipline).
- **Tombstone (deleted) entries:** a record with `length == 0` is a tombstone that shadows lower archives (RESEARCH `TreeFile_SearchNode.cpp:397-401`). Surface it as a distinct row state: `⊘` glyph (aria-label "Deleted (tombstone) — hides lower archives"), `--color-text-faint`, italic-free, with the chain still shown.

### States

| State | Content |
|-------|---------|
| Empty (no archive mounted) | Heading `No archive mounted` (`--text-sm`, `--color-text-muted`) + body `Mount Archive… to browse a .tre virtual filesystem` (`--text-xs`, `--color-text-faint`). *(updates the Phase-0 copy; see Copywriting)* |
| Mounting (in progress) | Inline progress affordance (see Surface 4) replacing the tree area: `Mounting {filename}…` + progress bar; toolbar disabled. UI stays responsive (CORE-06). |
| Mounted, empty search result | `No files match "{query}"` (`--text-sm`, `--color-text-muted`) + `Clear search` text action. |
| Mount failure | Error block (see Error states): `Could not mount {filename}` + reason + `Retry`. |

## Surface 2 — IFF FORM/Chunk Tree Viewer (BOTTOM / Data pane — new "Structure" tab)

When a `.iff` (or any IFF-framed) file is selected in the VFS browser, the BOTTOM data pane gains a
**Structure** tab presenting the parsed FORM/chunk tree (CORE-03). Format-agnostic (D-07): no per-format
typed decode in Phase 1.

### Tree node anatomy (each node = one IFF block)

A node is a row with these columns (the tree is effectively a two-pane: indented tag tree on the left,
aligned technical columns on the right; all technical values **mono `--text-xs`**):

| Column | Content | Notes |
|--------|---------|-------|
| Tag | 4-char tag, mono (e.g. `FORM`, `DERV`, `0001`) | container tags get a `▶` chevron + folder affordance |
| Kind | `form` / `leaf` (+ sub-type tag for forms, e.g. `FORM:SLOD`) | container vs leaf discriminator (RESEARCH: first-tag == `FORM`; generic viewer also recognizes `LIST`, `CAT ` as containers, PROP as leaf) |
| Size | declared payload length (mono, e.g. `0x0000_2A1C` / `10780`) | show hex + decimal; FORM length note: includes the 4-byte sub-type (RESEARCH) — surface the child-span the user expects |
| Offset | absolute byte-offset of the block header (mono, `0x…`) | this is what the hex pane jumps to on selection |

- Indentation: 16px per nesting level (inherited).
- Selection: 2px accent left border + `--color-accent-dim` (inherited tree-row selected state). Selecting a node **scrolls + highlights the corresponding byte range in the hex pane** (Surface 3).
- Container expand/collapse via chevron (rotates, 0.15s — inherited transition).

### Trailing-bytes & verification node (CORE-03 "zero unexplained trailing bytes")

- If the file has bytes after the last top-level block, the tree appends an explicit **`⟨trailing bytes⟩`** node (`--color-warn`, `▴` glyph, aria-label "Unexplained trailing bytes") showing `{n} bytes @ 0x{offset}` so they are **never silently dropped**. Selecting it highlights those bytes in the hex pane.
- If there are none, a footer status line reads `0 trailing bytes ✓` (`--color-accent`, mono).
- A footer **round-trip status** line reflects the harness gate for the loaded file: `byte-exact ✓` (accent) or `round-trip FAIL @ 0x{offset}` (danger). *(Status is computed by the harness/native side; the UI only displays it — D-08: no in-UI edit/save.)*

### States

| State | Content |
|-------|---------|
| No IFF selected | `No structure loaded` + `Select an IFF file in the Assets panel` (`--text-sm`/`--text-xs`, muted/faint). |
| Parsing (in progress) | `Parsing {filename}…` + indeterminate progress; pane stays responsive (CORE-06). |
| Parse error / malformed | `✕ Could not parse {filename} as IFF` (`--color-danger`) + reason (e.g. `invalid block length @ 0x{offset}`) + the raw bytes still openable in the hex pane (`Open raw bytes` action). |

## Surface 3 — Hex / ASCII Inspector Pane (BOTTOM / Data pane — "Hex" tab)

The SIE-successor baseline (D-07). Shows the raw bytes of the **selected IFF leaf chunk** (or the whole
file / a selected byte range) as a classic `offset │ hex │ ascii` grid. Read-only in Phase 1 (D-08).

### Grid contract

- Three regions per row, all **mono `--text-xs`**, row height 18px:
  - **Offset gutter:** 8-hex-digit absolute offset, `--color-text-faint`, right-aligned, `--space-2` right padding. Background `--color-header`.
  - **Hex columns:** 16 bytes per row, grouped 8 + 8 with a wider gap at the midpoint; two-hex-digit bytes, `--color-text`. Even/odd byte-column subtle shade via `--color-surface-2` (NOT color-meaningful — purely a reading aid).
  - **ASCII gutter:** 16 chars; printable bytes in `--color-text`, non-printable rendered as `·` in `--color-text-faint`.
- **Selected-range highlight:** the byte range of the currently selected IFF node header/payload is highlighted with `--color-accent-dim` across both the hex and ascii regions (the accent-reserved "selected hex-byte highlight"). Hovering a hex byte highlights the paired ascii char and vice-versa (`--color-surface-2`).
- **Header row** (sticky): column-offset ruler `00 01 02 … 0F` in `--color-text-muted`, mono, on `--color-header`.
- Virtualized rendering — only visible rows are in the DOM (multi-MB chunks must scroll smoothly; consistent with the CORE-06 responsiveness mandate).

### States

| State | Content |
|-------|---------|
| Empty | `No bytes to inspect` + `Select an IFF chunk or file` (muted/faint). |
| Loading bytes | brief skeleton/`loading bytes…` (zero-copy fetch via ArrayBuffer; should be near-instant for a single chunk). |

## Surface 4 — Async / Progress & Verification-Status Affordances (CORE-06)

Every heavy op (multi-GB mount, TOC/name-block inflate, large IFF parse, repack) runs off-main-thread
via `Napi::AsyncWorker` (RESEARCH-resolved). The UI must **never freeze** and must show progress.

### Progress affordances

| Context | Affordance |
|---------|------------|
| Archive mount/decompress | **Determinate** progress bar when total size is known: `--color-accent` fill on `--color-widget` track, 3px tall, with caption `Mounting {filename} · {pct}%` (mono `--text-xs`). Bar lives in the VFS browser body, replacing the tree until ready. |
| Large IFF parse | **Indeterminate** progress (animated accent sweep) with caption `Parsing {filename}…` — parse time is not size-linear; do not fake a percentage. |
| Repack / round-trip verify (test-driven, but if surfaced) | Indeterminate with caption `Verifying round-trip…`; resolves to `byte-exact ✓` / `round-trip FAIL`. |

- **Cancel:** mount and parse progress affordances include a `Cancel` text action (aria-reachable). Cancelling an async worker returns to the prior empty/selected state.
- **Responsiveness proof:** the status bar's existing live indicators (inherited from Phase 0) must keep ticking during a multi-GB mount — this is the visible proof that the main thread is not blocked. No spinner may be implemented in a way that blocks input (no synchronous native call on the JS thread).

### Status-bar additions (extend the Phase-0 status bar, do not redesign it)

Append to the inherited status bar (mono, `--text-xs`, `--color-text-muted`; success tokens in accent):

- `mount: [{n} archives]` — count of mounted archives; accent when ≥1.
- `vfs: [{n} files]` — total resolved entry count.
- During an async op: `[⟳ mounting 62%]` / `[⟳ parsing…]` (the `⟳` is the only live-op glyph; accent).
- `round-trip: [PASS]` / `[FAIL]` for the last verified file — PASS in accent, FAIL in `--color-danger`.

| Destructive confirmation template (reserved, unused this phase) | "[Action] [Target]?" + "This cannot be undone." + [Cancel] [Confirm verb] |

**Copy note on byte-exactness:** never claim "byte-exact" for a v0006 archive's payloads — they are
encrypted (D-05). Use "enumerate-only (structure verified)" for v0006; reserve "byte-exact ✓" for
v0005 payload round-trips and for self-built archive round-trips (RESEARCH OPEN-2 scope).
