<ground_truth>
<!-- VERIFIED against real client builder source. DO NOT re-derive. -->
- Builder write order (TreeFileBuilder.cpp:773-833): (1) 36-byte header with only token/version/numberOfFiles set; (2) file payloads in RESPONSE-FILE order (not crc/name order), each compressed-or-raw; (3) TOC (writeTableOfContents) entries in crc/name-sorted tocOrder, compressed if it helps; (4) name block null-terminated names in tocOrder; (5) MD5 block = numberOfFiles x 16 bytes, ALWAYS uncompressed; (6) seek back to 0, re-write the full header with the now-known toc/name offsets+sizes+compressors.
- MD5 block is written by the builder but NEVER read by SearchTree — a self-built deterministic re-build MUST reproduce it + response-file payload ordering + the builder zlib level.
- **zlib write level = `Z_DEFAULT_COMPRESSION` = level 6, default wbits 15 (RFC1950 framing, `78 9C`), default memLevel 8 — verified at `ZlibCompressor.cpp:169` (`deflateInit(&z, Z_DEFAULT_COMPRESSION)`).** Used ONLY when strictly smaller and input > 1024 bytes (compressAndWrite tries only CT_zlib; emits 0 or 2). PIN these exact parameters on the write path.
- **Deflate is NOT bit-stable across zlib versions/builds.** Therefore: self-built determinism = build-twice-with-OUR-writer identity (a regression guard, NOT "matches retail"); retail fidelity = copy untouched entries' raw compressed slices verbatim, recompress only edited entries (TreWriter.cs:166-174). DROP any "freshly compressed matches retail" claim.
- **miniz is FORBIDDEN on the WRITE path** (it cannot reproduce zlib's bitstream); the read path may use a miniz fallback. The writer links zlib and pins level 6 / wbits 15 / memLevel 8.
- TOC = CRC-first 6-field records; stride per version (from TreVersion.h recordStride, arbiter-confirmed — see Plan 01). Header = 36 bytes.
- v6000 payloads are encrypted -> NEVER written/round-tripped at the payload level; structural/enumeration only.
- CRC is computed over the fixUpFileName-normalized name (Crc::calculate).
</ground_truth>
