<objective>
Wire the Electron security posture and the complete utility-process to SAB to renderer pipeline, and prove zero-copy SHARING (not a value copy) with a same-memory cross-write driven by a per-run nonce. Main creates BrowserWindow with contextIsolation, registers COOP/COEP response headers synchronously before loadURL, forks the utility process, and relays the SharedArrayBuffer (over the transferred MessagePort) to the renderer. The preload exposes a narrow typed bridge. The renderer observes crossOriginIsolated === true, reads the C++-written 0xDEAD sentinel, then writes a per-run nonce into a second slot WITHOUT telling the utility that value over IPC; the utility re-reads that slot from the same held SAB and acks it back over the IPC demux route; the renderer asserts the echoed value equals its nonce.

Purpose: Proves SC-1 (security posture), SC-2 (hello round-trip), and SC-3/SC-4 (SAB sharing / cross-origin isolation), which is D-04. Round 1's plan proved only that a value survived the IPC hop — a serialized copy passes that test identically. The cross-write proof (review fix HIGH-3 / Opus + Cursor) is the only thing distinguishing genuine zero-copy sharing from a copy, and round 2 hardened it with a nonce so a copy-only utility that ECHOED the IPC arg can no longer false-pass (review fix MEDIUM-4 / Opus). Cursor's primary-source verdict (review fix HIGH-3): cross-process utility->renderer SAB sharing is UNDOCUMENTED and primary sources lean THROWS or COPY — SHARED is the unlikely outcome. This plan therefore treats the cross-write as a LIKELY-NEGATIVE de-risk experiment: it runs an EARLY canary, is NOT autonomous, and a failing cross-write BLOCKS (does not soft-pass) the live-sync claim, surfacing the documented architecture-pivot contingency.

Output: main.ts, preload.ts, utility-worker.ts delivering a running Electron app with the SAB channel live, the security posture locked, and a nonce-hardened cross-write proof (or an explicit, surfaced finding that cross-process SAB sharing is copy-only at the utility boundary, with the pivot contingency recorded).
</objective>

<architecture_finding>
## utility->renderer SAB sharing is a LIKELY-NEGATIVE experiment (review fix HIGH-3 / Cursor primary-source verdict)

Cursor's primary-source check (HTML structured-clone + agent clusters; Node SAB-sharing-between-threads-of-one-process;
Electron's documented IPC shape) concluded: serializing a SharedArrayBuffer reuses the same backing block ONLY within
one agent cluster; across different clusters it throws DataCloneError, with NO spec'd "clone into an independent SAB
copy" fallback. A crossOriginIsolated renderer is a web agent; an Electron utility process (separate OS child via
process.parentPort) is NOT in the renderer's cluster. So the likely boundary outcomes are THROWS (sab-ready relay fails)
or COPY (0xDEAD reads back but the cross-write fails); SHARED is the UNLIKELY outcome. D-04's premise depends on this
EXPERIMENT, not on an API guarantee — running it in Phase 0 is the right call, but it must be treated as likely-negative.

### Documented architecture-pivot contingency (record in the SUMMARY if the cross-write fails — do NOT rewrite D-02/D-04)
If the cross-write proves copy-only (or throws) at the utility->renderer boundary, live-sync must pivot. Candidate pivots,
in order of least disruption (this is a FINDING/risk note, not a re-decision of the locked CONTEXT.md decisions):
  1. MAIN-OWNED SAB: allocate/own the SAB in the MAIN process (same cluster path as the documented main<->renderer
     zero-copy patterns) and have the utility communicate deltas to main, main shares the SAB to the renderer.
  2. OS SHARED MEMORY (RESEARCH-IF-REACHED — demoted below option 1; round-3 / Sonnet SON-D): back the buffer with a
     Windows named file mapping (CreateFileMapping + MapViewOfFile via the native addon, or an npm shared-memory package
     such as a node-shared-memory/mmap-io style binding) that the utility and renderer-side native code both map,
     bypassing the structured-clone agent-cluster restriction entirely. This is a concrete mechanism but is LOWER
     priority than option 1 (main-owned SAB) — option 1 is the correct first pivot and is well-specified; option 2 is
     only researched if option 1 also fails.
  3. DROP THE UTILITY FROM THE HOT PATH: keep crash isolation for parsing, but run the 60fps SAB owner in main (accept
     reduced isolation for the hot channel only).
Per AGENTS.md, the zero-copy assumptions in docs/00-overview/architecture.md and docs/04-live-sync/live-memory-and-ipc.md
are marked UNVERIFIED-pending-this-experiment until the cross-write passes in dev (this plan) AND packaged (Plan 05).
</architecture_finding>
