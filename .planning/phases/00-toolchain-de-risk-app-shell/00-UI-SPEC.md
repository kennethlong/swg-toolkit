# Phase 0 — UI Design Contract: Toolchain De-risk & App Shell

> Visual and interaction contract for the Phase 0 app shell. Generated by gsd-ui-researcher.
> Consumed by gsd-planner (task decomposition), gsd-executor (implementation), gsd-ui-checker (validation).

**Scope:** The shell only — titlebar, four dock panels (left/center/right/bottom), status bar,
docking affordances, theme system, and the empty/seed states of each panel. No real SWG format
editors. Every concrete value below is sourced from the validated sketch 001 (Variant A) and its
CSS theme files unless noted otherwise.

## Spacing Scale — 3px DCC base grid (project grid exception)

This project deliberately overrides the standard 4px web grid with a 3px DCC base unit to match
Blender-style panel density (validated in sketch 001). All spacing tokens are multiples of the 3px
project base unit, not 4px. This is a documented, intentional project-level exception.

Use these tokens exclusively for intra-component and layout spacing. Do not use arbitrary pixel values.

| Token | Value | Usage |
|-------|-------|-------|
| `--space-1` | 3px | Icon-to-label gaps, dot indicators, chip internal padding |
| `--space-2` | 6px | Inline button gaps, tab padding, toolbar item spacing |
| `--space-3` | 9px | Accordion head padding (vertical), tree-row vertical padding |
| `--space-4` | 12px | Panel body padding, statusbar horizontal padding, section insets |
| `--space-6` | 18px | (reserved — inter-section spacing) |
| `--space-8` | 24px | (reserved — major layout gaps) |

**Exceptions:**
- Titlebar, tabstrip, statusbar heights are fixed chrome dimensions (`--titlebar-h: 30px`,
  `--tabstrip-h: 30px`, `--statusbar-h: 22px`, `--activity-w: 46px`) — these are not spacing
  values; they are chrome dimensions and must not be changed without re-doing the sketch.
- Workspace gap between panels: 5px (the sketch uses `padding: 5px; gap: 5px` on the workspace
  grid) — this is a structural grid value, not a spacing token.
- Drag-handle gutter (`.drag-dots`): 18px wide — fixed affordance width.

## Accessibility Contract

These are hard requirements (not suggestions), enforced by the UI checker.

### Rule 1: State must never be conveyed by color alone

Every selected/active/focus state uses a **combination** of border + background (or underline +
background). Color is additive, not the sole signal.

| State | Signal Combination | Token(s) |
|-------|--------------------|----------|
| Tree row selected | Left border 2px accent + background dim | `border-left-color: --color-accent` + `background: --color-accent-dim` |
| Tab active | Bottom underline 2px accent + surface background | `::after` 2px + `background: --color-surface` |
| Datatable row selected | Left inset box-shadow 2px accent + dim background | `box-shadow: inset 2px 0 0 --color-accent` + `background: --color-accent-dim` |
| Input focused | 2-layer focus ring (accent outline) | `box-shadow: var(--focus-ring)` |
| Activity rail active (Variant C) | Left 2px accent pip + accent-colored icon | `::before` left bar + `color: --color-accent` |

### Rule 2: Focus ring on all interactive elements

Every clickable/focusable element (numbox inputs, select, tabs, buttons, tree rows, table rows,
accordion heads, chips) must display `box-shadow: var(--focus-ring)` on `:focus-visible`. No
exceptions. The focus ring token differs between default and high-contrast themes (see above).

### Rule 3: Theme is a first-class, persistent, accessibility-grade feature

- The theme/accent picker lives in the **app titlebar** (not in settings/preferences), visible at
  all times, with label "Theme".
- Options: Hologram cyan (default) | SWG green | Amber | IDE blue | High contrast.
- Selection persists across restart (same mechanism as layout persistence — `localStorage` or
  Electron `app.getPath('userData')` store).
- Theme swap must re-style the entire app immediately with no flash/reload. Achieved by swapping
  a single `<link>` element's `href` (or a `data-theme` attribute + `@layer` CSS) that points to
  the appropriate theme CSS file.

### Rule 4: WCAG AA minimum text contrast

- `--color-text` (#e6e6e6) on `--color-surface` (#242424): contrast ratio ≈ 9.5:1 — PASS AA + AAA.
- `--color-text-muted` (#9a9a9a) on `--color-surface` (#242424): contrast ratio ≈ 4.7:1 — PASS AA (large text) / PASS AA for UI components per WCAG 3.0.
- High-contrast theme: white (#fff) on #101012 — PASS AAA.
- Accent `--color-accent-text` (#04201f) on `--color-accent` (#38bec9): contrast ratio ≈ 6.1:1 — PASS AA.

### Rule 5: Every icon-only / glyph-only control must carry an accessible name

Icon-only controls (panel action buttons, tab close, viewport tool chips, titlebar window controls)
convey meaning exclusively through a glyph. Every such element MUST declare both:

- `aria-label="<descriptive name>"` — read by screen readers
- `title="<same descriptive name>"` — visible tooltip on hover

Required labels for Phase 0 controls (exhaustive list — extend as new controls are added):

| Control | Glyph | `aria-label` / `title` |
|---------|-------|----------------------|
| Panel split button | ⊟ | `"Split panel"` |
| Panel collapse button (open) | ▾ | `"Collapse panel"` |
| Panel collapse button (closed) | ▸ | `"Expand panel"` |
| Panel maximize button | ⤢ | `"Maximize panel"` |
| Panel close button | × | `"Close panel"` |
| Tab close button | × | `"Close tab"` |
| Titlebar minimize | ● (yellow) | `"Minimize window"` |
| Titlebar maximize | ● (green) | `"Maximize window"` |
| Titlebar close | ● (red) | `"Close window"` |
| Viewport chip — Orbit | ⟲ | `"Orbit camera"` |
| Viewport chip — Pan | ✥ | `"Pan camera"` |
| Viewport chip — Frame | ⛶ | `"Frame selection"` |

This requirement applies equally in all themes, including high-contrast. The planner must create
a task for auditing every icon-only control for `aria-label` + `title` presence. See Panel Chrome
section below for the implementation note.

### Panel Chrome (all four panels)

Every panel is a flex column with:
- **Panel head** (`--tabstrip-h: 30px`): drag handle (18px, ⠿ glyph) + tab strip + action buttons
- **Panel body**: `flex: 1; overflow: auto; min-height: 0`
- **Border**: `1px solid --color-border` with `border-radius: --radius-md` (6px)
- **Background**: `--color-surface`
- **Shadow**: `--shadow-panel: 0 2px 8px rgba(0,0,0,0.35)`

**Tab strip:**
- Inactive tab: `color: --color-text-muted; background: transparent`
- Hover tab: `background: --color-header-2`
- Active tab: `background: --color-surface` + 2px bottom accent underline via `::after`
- Tab separator: `border-right: 1px solid --color-border`

**Panel action buttons** (split / collapse / maximize / close — 22x22px):
- Resting: `color: --color-text-faint; background: transparent`
- Hover: `background: --color-widget; color: --color-text`
- Focus-visible: `box-shadow: var(--focus-ring)`
- **Accessibility (Rule 5):** Every action button and icon-only glyph control MUST render with
  `aria-label` + `title` per the table in Rule 5 above. No icon-only button may be implemented
  without both attributes in place.

**Collapse behavior:** Clicking the collapse button (▾) hides `panel-body` and `panel-tabs`,
leaving only the panel-head visible. Button changes to ▸. Click again to restore.

### Docking Affordances

These are visual contracts for the dockview integration. The actual behavior is provided by

click targets.

**5-way drop zones** (shown during drag):
- 5 zones: center (tab), top-split, bottom-split, left-split, right-split
- Rendering: `background: --drop-fill` (18% opacity accent), `border: 1.5px dashed --drop-line`
  (85% opacity accent), `border-radius: --radius-sm`
- Hover on drop zone: accent fill at 32%
- Text hint: "drop to dock · center=tab · edges=split" in `--color-accent` on dark background

**Tab maximize button** (viewport only): ⤢ symbol, same 22x22 action button styling.

## Theme Persistence Contract

Both layout and theme must persist across restart. Same storage key pattern:

| Data | Storage Key | Storage Target |
|------|-------------|----------------|
| Dockview layout (JSON) | `swg-workspace-layout` | `localStorage` |
| Active theme name | `swg-active-theme` | `localStorage` |

On startup:
1. Read `swg-active-theme` from localStorage.
2. If present, swap the theme `<link>` element to that theme's CSS file before first paint.
3. Set the titlebar theme `<select>` value to match.
4. Read `swg-workspace-layout` and call `dockviewApi.fromJSON()` in the `onReady` callback.

If either key is missing (first launch), use defaults: theme = "cyan", layout = four-panel default.
